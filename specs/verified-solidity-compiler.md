# Feature: Standalone Verified Solidity Compiler (`reach-sol`)

## Feature Description
Extract the Reach compiler's Ethereum pipeline into a standalone "verified Solidity" workflow: a developer writes a contract as a Reach program, the compiler runs its Z3-based formal verification (token linearity, balance sufficiency, arithmetic overflow, assertion honesty under "ALL honest" and "NO honest" participant models, knowledge/privacy checks) over the optimized intermediate representation, and — only if verification passes — emits a self-contained, audited Solidity source file plus its ABI and a machine-readable verification report. The output drops into standard Ethereum tooling (Foundry/Hardhat/ethers) with no dependency on Reach's JS runtime, Docker images, or Algorand toolchain.

Today this capability exists but is buried: `reachc --intermediate-files` writes `<src>.<app>.sol` alongside a dozen debug artifacts, the ABI is only embedded inside the generated JS backend, verification results are unstructured stdout text, and compilation requires `goal` (the Algorand assembler) even for ETH-only work unless the source opts out. This feature turns that buried capability into a first-class, documented product surface.

## User Story
As a Solidity/EVM developer
I want to author smart contracts in a language whose compiler formally verifies them and then hands me plain Solidity + ABI artifacts
So that I can deploy mathematically checked contracts with my existing Ethereum toolchain (Foundry, Hardhat, ethers, viem) without adopting the whole Reach runtime stack.

## Problem Statement
Formal verification for Solidity is expensive and rare: auditors are costly, SMT tooling for raw Solidity (SMTChecker, Certora) is either shallow or proprietary, and most teams ship unverified contracts. Reach already contains a production-grade verifying compiler whose ETH backend emits Solidity — but it is only consumable as part of a full-stack DApp platform (JS runtime, Docker devnets, Algorand toolchain). There is no supported path to "just give me verified Solidity." Concretely:
1. No way to run an ETH-only compile without `goal` installed unless the `.rsh` source sets `connectors: [ETH]` (undocumented for this purpose).
2. The `.sol` file is an "intermediate file," emitted only with a debug flag, mixed with ~15 other debug artifacts.
3. The contract ABI is not emitted as a standalone artifact; it lives only inside `index.main.mjs`.
4. Verification results (theorem counts, failures, counterexample witnesses) are human-readable stdout only — nothing CI can consume.

## Solution Statement
Add a Solidity-output mode to `reachc` (`--sol`), plumbed through the existing `CompilerToolArgs → CompilerConfig` path, that:
1. Forces the ETH connector regardless of the source's `connectors` list (reusing the existing `dlo_connectors` mechanism from `Reach.AST.DL`), so ETH-only compiles never invoke `goal`.
2. Emits exactly three product artifacts into the output dir: `<src>.<app>.sol` (already generated by `Reach.Connector.ETH_Solidity`), `<src>.<app>.abi.json` (extracted from `csrAbi` in `Reach.Connector.ETH_solc.CompiledSolRec`, which already holds the ABI as JSON text), and `<src>.<app>.verify.json` (new structured verification report).
3. Skips the JS backend emission (`backend_js` call in `Reach.Compiler.mkCompileProg`) when in `--sol` mode.
4. Adds a `scripts/reach-sol` wrapper + `make reachc-dist` target so the compiler (plus its two runtime prerequisites, `z3` and `solc`) can be distributed and used standalone.

Verification semantics are untouched: the existing `Reach.Verify.verify` pass already runs before connector codegen and dies on failure (`maybeDie`), so `--sol` output is verified-or-absent by construction. The new report is a serialization of results the SMT layer already computes, not a change to what is checked.

## Implementation Plan
### Phase 1: Foundation
Add the flag surface and config plumbing with no behavior change: `co_solOnly`/`co_verifyReport` in `CommandLine.hs`, threaded through `reachc/Main.hs` into `CompilerConfig` (new fields `ccSolOnly`, `ccVerifyReport`). Define `Reach.VerifyReport` types with JSON instances and unit-test the serialization shape. Everything compiles with the flags unused; full golden suite must stay 806/806.

### Phase 2: Core Implementation
Wire the behavior in `mkCompileProg`: (a) when `ccSolOnly`, restrict `dlo_connectors` to the ETH connector (fail with a clear user error if the source explicitly excludes ETH); (b) skip `backend_js`; (c) mark the `.sol` output as a product output (always written to `outd`, not gated on `--intermediate-files` — reuse the `ccOutput opt` mechanism with `opt = True`); (d) emit `.abi.json` from `csrAbi` in the ETH connector's `conGen`. Extend `Reach.Verify.verify` to thread a result accumulator (an `IORef [VerifyTheoremResult]` in `VerifyOpts`, populated where SMT results are currently printed) and serialize it to `<src>.<app>.verify.json` after `maybeDie` — report written on success and, best-effort, on failure before dying.

### Phase 3: Integration
`reach sol` subcommand in the user CLI, `scripts/reach-sol` wrapper, `make reachc-dist`, the Foundry example under `examples/verified-solidity/`, the docs guide, golden tests for both modes, and AGENTS.md/manifest.yml registration.

## Testing Strategy
### Unit Tests
- JSON shape test for `Reach.VerifyReport` (field names are a public CI contract — lock them).
- CommandLine parser test: `--sol` sets `co_solOnly`; `--sol` + `--intermediate-files` compose.

### Integration Tests
- Golden `t/y/sol_only.rsh`: full verified ETH-only compile, stdout snapshot stable.
- Golden `t/n/sol_only_vfail.rsh`: verification failure exits nonzero, witness appears in stdout, no `.sol` emitted.
- Full existing suite (806 tests) as the regression net — `--sol` off must be byte-identical everywhere.
- Manual/CI-optional: Foundry deploy test of the emitted `.sol` in `examples/verified-solidity/`.

### Edge Cases
- Source with `connectors: [ALGO]` only + `--sol` → clear user error, not a crash.
- Multiple `Reach.App` exports in one file → one artifact triple per app (`<src>.<app>.*` naming already disambiguates).
- Verification timeout (`--verify-timeout`) in `--sol` mode → report records the timeout as a failure kind.
- `--sol` with verification disabled via the `REACH_ACCURSED_UNUTTERABLE_…` env var → refuse: `--sol`'s whole contract is verified-or-absent (check `ccShouldVerify` and error out).
- Contract exceeding the 24576-byte EVM limit (existing `maxContractLen` check in `ETH_solc.hs`) → error propagates, no partial artifacts.
- Output dir collision with a previous run → artifacts overwritten atomically (write-to-temp-then-rename), never a stale `.sol` next to a fresh `verify.json`.

## Acceptance Criteria
- `reachc --sol -o out/ prog.rsh` on a passing program emits exactly `prog.<app>.sol`, `prog.<app>.abi.json`, `prog.<app>.verify.json`, exits 0, and never executes `goal`.
- The same invocation on a failing program exits nonzero, prints the existing counterexample output, and leaves no `.sol` in `out/`.
- `verify.json` includes: total theorem count, per-honesty-mode results, and for each failure a source location and witness text; shape covered by a test.
- The emitted `.sol` compiles standalone with the pinned solc (0.8.26, `evmVersion: paris`) via `solc --standard-json` outside the Reach tree.
- Existing golden suite: 806/806 with zero accepted changes (i.e., default-mode output is untouched).
- `reach sol` subcommand works with `REACH_DOCKER=0`; `scripts/reach-sol` is shellcheck-clean.
- Docs page exists and AGENTS.md/manifest.yml reference the new mode.

## Notes
- **Authoring model honesty**: this feature does NOT verify arbitrary hand-written Solidity. The verification runs on Reach's intermediate representation; Solidity is the verified *output*. Marketing/docs must say "write verified contracts that compile to Solidity," not "verify your Solidity." A true Solidity frontend (parsing .sol into Reach IR) is a research-scale project — listed as future work, out of scope.
- **High-risk areas** touched: `hs/src/Reach/Verify/`, `hs/src/Reach/Connector/` — per repo policy these changes require human review; the plan deliberately restricts them to observation/artifact-emission so the SMT encoding and Solidity codegen are byte-identical.
- The `evmVersion: "paris"` pin (added during the solc 0.8.26 upgrade on this branch) is load-bearing for this feature: it makes the emitted `.sol`/bytecode deployable on pre-Shanghai and alt-EVM chains. Document it in the guide.
- `uv add` does not apply — this is a Haskell/stack project; any new Haskell dependency goes in `hs/package.open.yaml` extra-deps/`stack.yaml` (none are anticipated: `aeson` is already a dependency for the JSON report).
- Future considerations: (a) SARIF output for the verification report so GitHub code scanning can ingest it; (b) `--sol` equivalents for other connectors (`--teal`); (c) publishing `reachc-dist` artifacts from CI; (d) a Foundry plugin that wraps `reach-sol` as a build hook.
- Distribution reality check: standalone use still requires `z3` and `solc` binaries at the `DEPS`-pinned versions; the wrapper enforces this rather than hiding it.
