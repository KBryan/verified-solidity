## Step by Step Tasks
IMPORTANT: Execute every step in order, top to bottom.

### 1. Preflight
- `docker info` must succeed before ANY test or accept run. If Docker is down, `make hs-test-accept` silently corrupts `t/n` goldens (reachc's `goal` failure text gets accepted as expected output — this happened once already and was reverted).
- Confirm the Docker-backed `goal` shim is on PATH and functional: it must mount the repo and TMPDIR at identical container paths (`/private/var/folders` both spellings, `/private/tmp`) AND pass `-w "$PWD"`. A shim missing either half fails 450+ tests.
- `cd hs && make expand` — fast sanity check. If `mo` crashes under macOS bash 3.2 it leaves the target file EMPTY and make then treats it as up to date; `rm` the target before retrying (patched mo in session scratchpad `bin/`, or `brew install bash`).

### 2. Verify the paris fix (deploy smoke test round 2)
- Recompile `examples/overview` locally: `REACH_DOCKER=0 ./reach compile examples/overview/index.rsh`.
- Repeat the round-1 smoke test that caught the PUSH0 failure: ETH devnet via the published `runner:0.1.13` images + docker-compose shim, deploy `examples/overview`. Round 1 died at `eth_estimateGas` with `invalid opcode: PUSH0`; with `evmVersion: "paris"` the deploy must succeed.
- Independent bytecode check: the emitted init code must not contain the PUSH0 opcode where shanghai codegen used it (compare against a scratch compile with the paris line commented out if in doubt).
- Run the Foundry deploy test for the verified-solidity path: `examples/verified-solidity/foundry-test/run.sh` (forge is installed locally; it passed pre-paris — confirm still green).
- Record results in `.agent/baseline.md` (Phase 4 section, "round 2" entry).

### 3. Land hop 1 + paris fix as shippable commits
- Audit `git diff hs/t/y/tooBig.txt hs/t/y/gh-1183.txt` — size integers only; `git diff hs/t | grep -i docker` must be empty.
- Commit A (stack hop): `hs/stack.yaml` + `hs/stack.yaml.lock` only — resolver lts-20.26, dropped snapshot-covered pins, tomland/validation-selective re-pins. Message notes: zero source changes, 806/806 green under GHC 9.2.8.
- Commit B (paris fix, FLAG FOR HUMAN REVIEW): `hs/src/Reach/Connector/ETH_solc.hs` + the two re-accepted goldens. Message states the PUSH0/shanghai regression, the smoke-test evidence, and that this is a high-risk connector change.
- Do NOT fold unrelated working-tree changes (verified-solidity feature files, agent config) into these commits — they are separate work streams on this branch.
- End commit messages with the `Co-Authored-By: Claude Fable 5 <noreply@anthropic.com>` trailer.

### 4. Fix `make hs-format` (ormolu gfork pin)
- Diagnose: `cd hs && stack build ormolu` — fails because the gfork commit's stack extra-deps (`ghc-lib-parser 8.10.7`, `optparse-applicative 0.16.1`) predate lts-20.26.
- Determine what the gfork actually changes vs upstream ormolu (`git log`/diff of `github.com/google/ormolu` branch `gfork` against its merge-base with upstream). Decision tree:
  - If the fork delta is formatting-irrelevant or upstream ormolu ≥0.5 (in lts-20.26's snapshot) produces an identical result on `hs/src/`: drop the git pin from `stack.yaml` and use the snapshot's ormolu. Verify by running `make hs-format` and checking `git diff --stat hs/src hs/app` — a zero or trivial diff confirms; a large reformat means the fork delta mattered.
  - If the fork delta matters: keep the pin and add the resolver-compatible extra-deps it needs, OR bump the pin to a gfork rebase if one exists.
  - If neither is tractable this pass: pin ormolu as a standalone binary install (brew) and change `hs-format` to use it — last resort, note it in baseline.md as debt.
- If dropping/changing ormolu version causes a mechanical reformat of the tree, commit that reformat as its own isolated commit (no logic changes mixed in).
- Validate: `make hs-format` exits 0; follow-up `git diff` after a second run is empty (idempotent).

### 5. Hop 2: lts-20.26 → lts-21.25 (GHC 9.4.8)
- Edit `hs/stack.yaml` resolver; run `stack build` and follow its pin suggestions: check whether `tomland`/`validation-selective`/`io-capture`/`simple-smt` pins are now in (or still out of) the snapshot; drop or re-pin per stack's errors.
- Expected GHC 9.4 friction: `mtl`/`transformers` re-export tightening (add explicit `Control.Monad` imports), `Data.List` singleton/type-ambiguity warnings, possibly `-Werror` warning churn. Fix forward — mechanical import/pragma changes only; anything semantic in `Connector/` or `Verify/` stops and gets flagged.
- The five git forks are the risk (`language-javascript` reach-1 is the parser core — a compile failure there blocks everything). If a fork fails under GHC 9.4: prefer bumping the pin to an upstream commit that supports 9.4 IF the reach delta is preserved; otherwise patch the fork commit reference. Do not silently swap `language-javascript` to upstream — the reach-1 branch carries parser changes reachc depends on.
- Validate (full ladder): `make expand` → `make hs-build` → `make hs-test` (expect 812/812 with the verified-solidity tests in-tree; 806 if testing a tree without them). Golden drift at this hop should be ZERO (compiler version bumps don't change codegen); any golden diff is a red flag, not an accept candidate.
- Commit: stack.yaml(+lock) + any mechanical source fixes, one commit, message records test counts.
